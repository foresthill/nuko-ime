## 1. コンテクスト予測変換

### 1.1 一文単位のコンテクスト
入力中の一文全体を見て、最適な変換を選ぶ。
今のIMEは「一語ずつ確定」「一文節ずつ確定」が多いが、これを文単位に拡張する。

### 1.2 段落単位のコンテクスト (前段落・後段落も判断材料)
- 前後の段落も加味して、その文の変換候補を決める。
- 例: 仕事の文書を書いている文脈なら「あつい」→「熱い」より「暑い」より「篤い」を選ぶ判断ができる、など (要は意味的整合)。
- どこまでメモリ保持するか、どこまでスキャンするかが設計上の論点。

### 1.3 長いコンテクスト
- 数千文字オーダーで context window を保持し、文書全体の意味整合をとる。
- LLMライクなアプローチも視野。ただしオフラインで動かせる軽量モデルである必要がある。

## 2. 優先順位ロジック (重み付け)

### 2.1 設計原則
- **単純** (デバッグ可能で挙動が読める)
- **クリア** (ユーザーが「なぜこの候補が来たか」を直感できる)
- **スマート** (個別ユーザーに最適化される)

### 2.2 ユーザー固有の傾向性
「このユーザーはこの言葉をよく使う」を学習し、優先順位に反映。
単に頻度カウントするだけでなく、**文脈との整合性も加味した重み付け**。

### 2.3 想定される素朴な式
```
score(候補, 文脈) =
    α × 頻度(ユーザー)
  + β × 文脈整合(前後の単語/段落との関係)
  + γ × 辞書スコア
  + δ × 直近の使用 (recency)
```
α, β, γ, δ をユーザーが調整可能、もしくは自動チューニング。

## 3. 候補選択 UX の刷新

### 3.1 現状の問題
- macOS / Windows IME 標準は **Space / 矢印キー / Tab で候補移動** という Windows 95 時代の UI。
- 一度確定されると違う変換でも消して打ち直し → ストレスフル。

### 3.2 目指すもの
- **Windows 95 の "最適解" は現代では古い**。モダンなインタラクションを模索する。
- 候補ウィンドウのレイアウト、ホバー、ジェスチャー、ホットキーなどを再考。
- 確定が暴発しない、戻れる、迷わない UX。

### 3.3 アイデアの方向性 (要検討)
- 候補をリストではなくグリッド表示し、視認性を上げる
- 候補に対して理由を表示 (「過去使用回数」「文脈マッチ」等)
- 確定タイミングをユーザー操作起点で完全制御 (auto-commit を排除)
- マウスカーソル追従式のフローティング候補
- スマホIMEのフリック的な発想を PC で

### 3.4 設計哲学: モードを足して最適解を探す

二項対立で「続け打ち vs 確認重視」を選ぶのではなく、複数モードを共存させて
ユーザーがどれを使うか観測しながら最適解に収束させる (Vim 的進化)。

例:
- v0.1: 続け打ち (デフォルト、現行)
- v0.2: 文節アドレッシングモード (ブレスト #1、下記)
- v0.3: 信頼度自動委譲モード
- v0.4: 音声入力後処理モード (Section 7)

使われないモードは将来 deprecate、使われるモードは洗練。

### 3.5 ブレスト #1: 文節アドレッシング (パーキング中)

**アイデア**: ホームポジション (asdfghjkl) を文節にマップ。
変換後、各文節の上に小さく `a/s/d/...` ラベル表示。文節キーを押すと
その文節にフォーカス、候補巡回や修正ができる。全部終わったら Enter。

**強み**:
- 空間マッピングが直感的 (左右の文節 ↔ 左右のキー)
- モード切替不要 (キーが常に live)
- 指の移動ゼロ (タッチタイピング親和)

**派生案**:
- 修飾キー (Alt 等) で「アドレッシングモード」を一時起動 → 続け打ちと共存
- アルファベット 26 文字で 9 文節超に対応
- 信頼度高い文節は自動確定、低い文節だけアドレッシング対象

**主な弱点 (旧パーキング理由)**:
- **境界変更との相性が悪い**: 現行は Shift+Left/Right で文節境界を伸縮できるが、
  アドレッシング方式だとキー割当が破綻する。動詞を増やすほどシンプルさが死ぬ
- **続け打ちとの衝突**: 「a」が文節1選択なのか「あ」入力なのかの曖昧性
- **9文節以上の長文への拡張**

### 3.6 ブレスト #2: ゲームコントローラー流 Tap/Hold UX

ブレスト #1 のパーキング理由 (境界変更が困難) を解決する案。
任天堂のコントローラー設計の語彙を IME に持ち込む。

**Tap vs Hold + Press の使い分け**:

| ゲーム動詞 | 検出 | IME での意味 |
|-----------|------|-------------|
| Tap (短押し) | keydown→すぐkeyup、他キーなし | `a` → 文節1 を選択・候補巡回 |
| Hold + Press | `a` 押下中に別キー | `a + →` → 文節1 の境界右拡張 / `a + ←` → 縮小 |
| Hold + Release | `a` 押下→他キーなし→`a` 離す | 文節1 を確定して次へ |
| Double tap | 短時間2連打 | 文節1 を初期化して再変換 |
| Long press | 500ms以上保持 | 文節1 の編集モードへ |

**強み**:
- 動詞が分裂せず、修飾子で意味を増やす設計 (ABXY+方向キーで膨大な操作を成立させるゲーム手法)
- ブレスト #1 の境界変更問題が `a + ←/→` Hold combo で自然に解決
- Tap だけでも基本動作は完結 → 入門容易

**任天堂的に学びたい原則**:
1. **発見性**: ボタン押下時に視覚アニメーション (文節がハイライト) で学習を誘導
2. **猶予 (forgiveness window)**: Tap vs Hold の閾値に体感的なスラック (~250ms?)
3. **層状の複雑さ**: 初心者は Tap、上級者は Hold combo
4. **触覚/聴覚フィードバック**: 境界変更時の微振動・効果音 (ON/OFF設定可)
5. **入力の即応性**: 1フレーム遅延も妥協しない

**課題**:
- Tap/Hold 検出のための入力イベントモデル設計
- macOS IMK / Windows TSF での実装可否
- 既存 IME ユーザーへの教育コスト

## 4. さかのぼり再変換

- 一度確定された文章を、再度変換可能な状態に戻せる。
- 「変換確定が違った！」と気付いたとき、戻って消して打ち直す必要をなくす。
- 過去のIMEで類似機能を見たことがある (具体名は未確認、要調査)。

### 4.2 想定インタラクション

## 5. 入力ミスの自動補完

### 5.1 機能概要
- よくある typo を補完する
- 例: `くだしあ` → `ください` (sa と si の隣接ミス + a 重複等)
- 過去にこの機能があった IME を見たことがある (要調査、Google日本語入力にあったかも)

### 5.1.1 なぜ「ください → くだしあ」が起きるか (認知現象)

これは個人の癖ではなく、タイピング研究で広く知られた **転置エラー** の一種。
特に「**先取りエラー** (anticipatory error)」: 後ろの文字を脳が先に出してしまう。

- "kudasai" の場合: 末尾の `i` が脳内で先に発火 → `sa` の `a` と入れ替わって "kudashia" 化
- 日本語ローマ字では `sa` より `shi` の使用頻度が高く、筋肉記憶が `sh` 系に流れやすい
- 誰でも起こるが、頻度は個人差あり (= 個別学習が効く領域)

→ IME が補完する価値が高い。万人共通の典型 typo + 個別学習の二段構成が筋。

### 5.1.2 「ヘンコウ → ヘンコク」(挿入エラー)

転置とは別の **挿入エラー** パターン。`henkou` を打とうとして
`o` の後に余計な `k` が入って `henkoku` になる。

仮説:
- 末尾「〜こく」(一刻、報告、変革等) と「〜こう」(変更、考慮等) が脳内で競合
- どちらの語尾になるか曖昧な瞬間、`k` を保険的に出してしまう
- 個人差が大きい (どの語ペアで競合するかは人による) → 個別学習が効く

### 5.4 訂正イベント学習 (高優先度)

ユーザーが誤入力を直したイベント自体を最高品質の教師データとして使う。

**なぜ強いシグナルか**:
- 推測ではなく確定情報 (ユーザーが明示的に修正した)
- 文脈付き (前後の文をそのまま訓練データ化)
- 個別最適化に直結 (個人の癖は本人にしか分からない)

**検出ロジック (案)**:
```
trigger: insertText 直後 (5秒以内) に Backspace 連打 → 同じ範囲を再入力
record: { 元入力, 訂正後, 文脈±20文字, タイムスタンプ }
N回以上の閾値で自動補完候補に昇格
```

**安全装置**:
- 信頼度しきい値 (誤訂正の偶発的学習を排除)
- ユーザー設定で「学習をオフ」可能
- 学習履歴を可視化、削除可能 (透明性)

**派生効果**:
- 単なる typo 補完だけでなく「変換選好」も学習可能
  (例: ユーザーが常に「変換」より「返還」を選ぶならスコアに反映)

### 5.2 課題
- **エッジケース大量**: 「補正したくない時の」誤補正のリスク
  - 一覧化しきれない typo パターン
  - ユーザー意図を誤読してかえってストレスになるケース
- 補正を提示する/即適用の閾値設計
- 補正候補と通常候補を同時表示 (ユーザーが選べる) のが安全か

### 5.3 想定アプローチ
- レーベンシュタイン距離で近い既知単語を検出
- ユーザー履歴で頻出ミス → 個別補正学習
- 補正は「候補リストに追加する」だけ。**自動置換は絶対にしない**

  - 独自性は「カスタマイズ機構そのもの」と上位レイヤーで出す方針

## 7. 音声入力統合

### 7.1 背景
音声入力は macOS / iOS / Android で増加傾向。現状は End-to-End で
音声 → 漢字混じり文 まで一気に処理し、IME はバイパスされる。
これだと誤った漢字に変換されても普通の文字編集でしか修正できない。

### 7.2 音声フローと IME の関与
| ステップ | 内容 | IME の役割 |
|---------|------|-----------|
| A | 音声 → ひらがな (音声認識) | 関与しない (Whisper / Apple SR 等) |
| B | ひらがな → 漢字混じり文 (変換) | **IME の本来の仕事** |
| C | 結果に対する後修正 | **IME の再変換 (Section 4) と直結** |

### 7.3 ぬこIME 独自のポジション (案)

**D. 音声入力後の「再変換ファースト」UX**
- 音声で入力されたテキストの上に **再変換可能フラグ** を埋め込む
- 後でいつでも候補切替で fix できる
- 現状どの IME も提供していない領域

**E. 音声 + 文字の混合入力**
- 本文は音声で爆速入力
- 固有名詞・専門用語は手で打つ (音声認識が弱いところ)
- IME がモードを自動切替: 音声中は受動、手入力中は能動

### 7.4 関連
- Section 4 (さかのぼり再変換) と密接 — 音声誤認識の事後修正は再変換そのもの
- ローカル音声認識モデル (Whisper.cpp 等) との統合余地
- プライバシー観点: クラウド音声認識の代替提示
