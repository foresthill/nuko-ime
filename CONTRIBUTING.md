## 開発環境のセットアップ

### 前提条件

- Git
- Rust 1.75.0以上（rustup推奨）
- Python 3.11以上（ツール用、オプション）

### プラットフォーム固有の依存関係

#### Windows
- Visual Studio Build Tools 2022以上
- Windows SDK

#### macOS
- Xcode Command Line Tools
- macOS 11.0以上

#### Linux
- GCC 11以上 または Clang 14以上
- IBus開発ライブラリ (`libibus-1.0-dev`)

### セットアップ手順

1. リポジトリをフォークする

2. ローカルにクローンする
   ```bash
   git clone https://github.com/<あなたのユーザー名>/nuko-ime.git
   cd nuko-ime
   ```

3. Rustツールチェインを確認する
   ```bash
   rustup show
   # rust-toolchain.toml により自動的に適切なバージョンが選択されます
   ```

4. 開発ブランチを作成する
   ```bash
   git checkout -b feature/my-new-feature
   ```

5. ビルドする
   ```bash
   # 開発ビルド
   cargo build

   # リリースビルド
   cargo build --release
   ```

6. テストを実行する
   ```bash
   cargo test
   ```

7. CLIツールを試す
   ```bash
   # 変換テスト
   cargo run -p nuko-cli -- convert にほんご

   # ローマ字変換
   cargo run -p nuko-cli -- romaji nihongo

   # 情報表示
   cargo run -p nuko-cli -- info
   ```

## コントリビューションプロセス

### 課題の選択

1. [Issues](https://github.com/your-org/nuko-ime/issues)ページから取り組みたい課題を選ぶ
2. コメントして作業開始の意思表示をする
3. 未解決の課題に取り組む場合は、事前に議論を開始する
4. `good first issue` タグは初心者向けの課題です

### 変更の提出

1. 変更に対応したブランチを作成する
   - 機能追加: `feature/機能名`
   - バグ修正: `fix/バグ内容`
   - リファクタリング: `refactor/内容`
   - ドキュメント: `docs/内容`

2. 変更を加え、適切なコミットメッセージを付ける
   ```bash
   git commit -m "feat: ○○機能の追加"
   ```

   コミットメッセージのプレフィックス:
   - `feat`: 新機能
   - `fix`: バグ修正
   - `docs`: ドキュメント
   - `refactor`: リファクタリング
   - `test`: テスト追加・修正
   - `chore`: ビルド・ツール関連

3. 変更をフォークにプッシュする
   ```bash
   git push origin feature/機能名
   ```

4. プルリクエストを作成する
   - テンプレートに従って情報を記入
   - 関連するIssueへの参照を含める

### レビュープロセス

1. 少なくとも1人のコアメンテナからのレビューが必要
2. CI/CDチェックをパスする必要がある
3. レビューコメントに対応し、必要に応じて変更を加える
4. すべての条件が満たされたらマージされる

## コーディング規約

### 全般

- 命名は明確で意図が伝わるものにする
- コメントは日本語または英語で記述可能（ただし一貫性を保つ）
- 公開APIには適切なドキュメントコメントを付ける

### Rust

Rustの公式スタイルガイドおよびClippyの推奨に従います。

#### 命名規則
- 型名（構造体、列挙型、トレイト）: `PascalCase`
- 関数名、メソッド名: `snake_case`
- 変数名: `snake_case`
- 定数: `SCREAMING_SNAKE_CASE`
- モジュール名: `snake_case`
- ライフタイム: `'lowercase`

#### フォーマット
```bash
# コードフォーマット（必須）
cargo fmt

# Lintチェック（必須）
cargo clippy --all-targets --all-features -- -D warnings
```

#### ドキュメントコメント
```rust
/// 関数の説明
///
/// # 引数
/// * `param` - パラメータの説明
///
/// # 戻り値
/// 戻り値の説明
///
/// # エラー
/// エラーが発生する条件
///
/// # 例
/// ```rust
/// let result = my_function(42);
/// ```
pub fn my_function(param: i32) -> Result<String> {
    // ...
}
```

#### 推奨事項
- `#![forbid(unsafe_code)]` を可能な限り使用（OS統合部を除く）
- `#![warn(missing_docs)]` を有効にする
- エラーハンドリングには `thiserror` / `anyhow` を使用
- 公開APIには `#[must_use]` を適切に付与

### Python（ツール用）

- PEP 8に準拠
- 型ヒントを使用
- フォーマッタ: `ruff format`
- リンター: `ruff check`

## テスト要件

### 必須テスト

- すべての新機能には対応するユニットテストを追加
- バグ修正には再現テストを追加
- 公開APIの変更にはドキュメントテストを追加

### テスト実行

```bash
# 全テスト実行
cargo test

# 特定のクレートのテスト
cargo test -p nuko-core

# 特定のテスト
cargo test test_romaji_conversion

# ドキュメントテスト
cargo test --doc
```

### ベンチマーク

```bash
# ベンチマーク実行
cargo bench

# CLIベンチマークツール
cargo run -p nuko-cli --bin nuko-bench
```

### コードカバレッジ

- 新規コードは70%以上のカバレッジを目指す
- コアモジュール（nuko-core）は80%以上のカバレッジを維持

```bash
# カバレッジ計測（cargo-tarpaulin使用）
cargo tarpaulin --out Html
```

## ドキュメント

### コードドキュメント

- 公開API、構造体、関数には必ずドキュメントコメントを付ける
- rustdocフォーマットを使用する
- 複雑なロジックには説明コメントを付ける

```bash
# ドキュメント生成
cargo doc --open
```

### ユーザードキュメント

- 新機能にはユーザーマニュアルの更新を含める
- スクリーンショットやGIFなどの視覚的な説明を推奨
- 平易な言葉で説明し、専門用語は避けるか説明を付ける

### 開発者ドキュメント

- 設計決定の理由を記録する
- アーキテクチャの変更には図表を含める
- API変更はCHANGELOGに反映する

## リリースプロセス

### バージョニング

[セマンティックバージョニング](https://semver.org/lang/ja/)に従います：

- メジャーバージョン: 互換性を破る変更
- マイナーバージョン: 後方互換性のある機能追加
- パッチバージョン: 後方互換性のあるバグ修正

### リリース前チェックリスト

- [ ] すべてのテストがパスすること
- [ ] `cargo clippy` で警告がないこと
- [ ] `cargo fmt --check` でフォーマットが正しいこと
- [ ] ドキュメントが更新されていること
- [ ] コードレビューが完了していること
- [ ] CHANGELOGが更新されていること
- [ ] バージョン番号が適切に更新されていること

### リリースブランチ

1. `release/vX.Y.Z`ブランチを作成
2. 最終テストとレビューを実施
3. 問題がなければメインブランチにマージ
4. リリースタグを付ける
